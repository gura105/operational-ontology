# Operational Ontology

[![CI](https://github.com/gura105/operational-ontology/actions/workflows/ci.yml/badge.svg)](https://github.com/gura105/operational-ontology/actions/workflows/ci.yml)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](./LICENSE)

> **Operational Ontology（操作できるオントロジー）とは、自分が所有していないシステム群のデータの上に構築される共有ドメインモデルである — オブジェクト・リンク・アクションで構成され、読み取りはモデルを辿り、書き込みは業務ルールを持つアクションだけを通り、監査され、その変更が触れる状態を所有するシステムオブレコードへ書き戻される。**
>
> セマンティックレイヤーはビジネスを「読む」ためのもの。Operational Ontology はビジネスを「動かす」ためのもの。

Palantir Foundry の Ontology はこのパターンの一実装です。このリポジトリはもう一つの実装 — 最小で可読なリファレンス実装、つまり概念の最小化です。これはフレームワークではなく「動く定義」です。読んで、フォークして、アイデアを盗んでください。

## なぜ別の名前が要るのか

「オントロジー」という言葉があまりに多くの仕事を掛け持ちしているからです。哲学では存在するものの研究。OWL/RDF では機械が推論できる形式意味論。ナレッジグラフはエンティティと関係のグラフの意味で借用しました。そして 2026 年、大手データプラットフォーム各社が「オントロジー」を冠したコンテキストレイヤーを出荷し始めました — AI の回答を業務の意味で接地する、確かに有用な、しかし徹頭徹尾 read 側のものです。1 つの言葉に 5 つの意味。「オントロジー」を巡る会話は静かに壊れています。

| 「オントロジー」と呼ばれるもの | 実体 | 統制された書き込み |
| --- | --- | --- |
| 哲学のオントロジー | 存在するものの研究 | — |
| formal ontology（OWL / RDF） | 機械が推論できる形式意味論 | なし |
| ナレッジグラフ | エンティティと関係のグラフ — データとしては書けるが、操作としては書けない | なし |
| AI コンテキストレイヤー（2026 年の「オントロジー」ブランド製品群） | AI の回答を接地する意味づけ層 | なし |
| **Operational Ontology（Foundry 型）** | 業務ドメインのスキーマ **+ ルールを運ぶアクション** | **ある** |

どの行も正当な道具であり、この表は格付けではありません。しかし、層に*何ができるか*を変えるただ一つの性質 — 業務ルールに統制された書き込みを受け付けるかどうか — は表全体を横断する軸なのに、名前がありませんでした。このリポジトリはそれに名前を与えます。

## 4 つの性質

以下の 4 つがすべて成り立つとき、そのシステムはこのパターンを実装しています。4 つが定めるのは *what*（何が真でなければならないか）であって *how* ではありません — outbox か webhook か、SQL か検索インデックスか、ストアは 1 つか複数か: 機構はすべて実装の選択です。これは合格すべき検定ではなく、**システムについて議論するための共有語彙**です。

1. **セマンティックなオブジェクトとリンク。** 業務のエンティティとその関係が、先に存在していた物理データ — 他のシステムが所有するデータ — の上に明示的にモデリングされている。

2. **アクションに一本化された書き込み。** この層の状態が変わる理由は 3 つある: ソースが変わり、再インデックスがそれを反映する。モデル自体がレビューを経て進化する。あるいは、ビジネスが何かを決める。この性質が統治するのは 3 つ目である: **業務判断は名前のついたアクションを通してしか状態を変えない。** 汎用の更新経路は存在しない — ユーザーにも、アプリケーションにも、エージェントにも。残りの 2 つは抜け穴ではない: 再インデックスはソースが既に言っていることを再生するだけであり、スキーマ進化は「何を言えるか」を変えるだけで「何が真か」は変えない。**業務上の結果を選ぶ書き込みは、エンドポイントの名前が何であれ判断であり — 判断はアクションを通る。**

3. **アクションに紐づく業務ルール。** 事前条件はドメインの不変条件（「出荷済みの注文はキャンセルできない」）であり、機械可読なエラーで拒否される — アクセス制御でも UI バリデーションでもない。すべての試行は、適用も拒否も、監査ログに残る。

4. **システムオブレコードへの書き戻し（write-back）。** すべての書き込みには宣言された住所があり、どれがどれかはモデルが語る。**source-backed** な変更は、上流システムが所有する状態 — ERP がマスターする注文の status のような — に触れる。それは統制された順序つきの副作用としてそのソースへ書き戻される: 真実の所在は元の場所のまま。**ontology-owned** な変更は、どのソースシステムにもカラムがないもの — 担当者、トリアージメモ — を記録する。その状態にとっては、オントロジー自身のストアがシステムオブレコード*である*。偶然ではなく、宣言によって。（**derived** な状態 — 集計値、件数 — は計算されるもので、書き込まれることはない。）この性質が禁じるのは、ラベルのない中間地帯である: 故郷に帰ることのない、他システムの真実の影のコピー。誰も住所を言えない書き込み。そして source-backed な書き込みを 1 つも持たない実装は、このパターンの小さい版ではない。**自前のデータベースを持つただのアプリケーションが、語彙を着ているだけ**である。

リトマス質問: **「あなたのセマンティックレイヤーから注文をキャンセルできますか？」** 答えが No なら、それは読み取り層です — 有用ですが、別のもの。答えが Yes でも、どのシステムオブレコードの行も決して変わらないなら、それは並行データベース — これも別のもの。そして出荷済みの注文を文句も言わずキャンセルするなら、それは書き込み API です — 性質 3 が違いのすべてです。

### 実装が宣言すべきもの

パターンは機構については不可知ですが、沈黙については不可知ではありません。実装間で異なり、かつユーザーが観測できる選択がここにあり、実装はその答えを表明します。本リポジトリの答えは一つの実例です — 以下の斜体が要約で、機構的な詳細は[実装ノート](./IMPLEMENTATION.ja.md)にあります。（2 つ目の項目は[失敗時のセマンティクス](#失敗時のセマンティクス)が展開します: 鮮度・並行性・監査面はそのサブ回答です。）

- **Authority** — どの変更が source-backed で、どれが ontology-owned で、どの状態が derived か。*ここではモデルが二度答える: `owned` 宣言がオントロジー自身の所有する状態を印し — プロパティ（`Order.assignee`）、リンク型、あるいは型まるごと（`Note`） — アクションの `writeback: true` がその変更を source-backed と宣言する。宣言は信用されるのではなく検査される: ランタイムはすべての編集プランを `owned` 宣言に照らして分類し、アクションの宣言と矛盾するプランを拒否する（[詳細](./IMPLEMENTATION.ja.md#authority-の検査)）。Foundry はアクション単位の write-back webhook、edit-only プロパティ、宣言された競合解決戦略で答える。*
- **Failure semantics** — write-back とローカルコミットが食い違ったとき何が起きるか。*ここでは: write-back が先行し、ソースが拒否すればこちらは何も変わらない — [失敗時のセマンティクス](#失敗時のセマンティクス)参照。*
- **再インデックス × 編集** — ontology-owned な状態がベースの更新を生き延びるか。*ここでは: 生き延びる — Foundry と同じ形の最小化: ontology-owned プロパティへの編集はオーバーレイに住み、`load()` が新しいベースの上に再適用する。ontology-owned な型とリンクはそもそもスナップショットの手が届かない。編集を孤児にする再インデックスは全体が拒否される — [失敗時のセマンティクス](#失敗時のセマンティクス)参照。*
- **Visibility のデフォルト** — ポリシーのないオブジェクトが何に倒れるか。*ここでは: fail-open — [FAQ](#faq) 参照。*

本リポジトリと違う答えをしても、パターンの内側です — その答えこそが、する価値のある議論です。ある製品が Operational Ontology を名乗るなら、証明書を求めないでください。その答えを求めてください。

## クイックスタート

```sh
pnpm install
pnpm demo    # 物理データ → 統合 → インデックス → 読み取り → 書き込み → 拒否 → 書き戻し
pnpm test    # 振る舞いのテスト
```

デモの題材は、[このリポジトリが付属する記事](https://x.com/gura105/status/2077153028982133080)と同じシナリオです: ある企業が競合を買収し、**スキーマもステータスのエンコーディングも異なる 2 つのレガシー受注システム**を抱え込む。数十行の SQL と小さな TypeScript マッピングで統合し、その上に `Customer` / `Order` / `Product` — そしてどのソースシステムにもテーブルがない型 `Note` — をオントロジーとしてモデリングする。そして:

- リンク走査が「この製品を含む注文はどれか？」に両システム横断で答える
- `assignOrder` は**どのレガシーシステムにも存在しない状態**を書き込む — 編集はソースの上の層に住める
- 出荷済み注文への `cancelOrder` は `SHIPPED_ORDER_CANNOT_BE_CANCELLED` で**拒否される**
- 未出荷注文への `cancelOrder` は成功し、レガシー ERP の行が**実際に書き換わる**
- パイプラインが動いているレガシーシステムを**再インデックス**する: source-backed な状態は ERP から更新され、担当者とメモ — ontology-owned — は**生き延びる**
- すべての試行が — 適用も拒否も — 監査ログに残る

## AI エージェント向け（MCP）

```sh
pnpm mcp     # 同じオントロジーを stdio 経由でエージェントに公開
```

stdio ではすべての呼び出し元が 1 つの identity に潰れます。`OO_AGENT=<name> pnpm mcp` でこのサーバーが仕えるエージェントに名前を付けられます — これはラベリングであって、認証ではありません。

MCP のツール群は*モデルから生成*されます: `search_order`、`traverse_customer_orders`、`cancel_order`、`read_audit_log`…… クエリの形とアクションごとに 1 ツール。注目すべき点が 2 つあります:

- **raw SQL ツールは意図的に存在しません。** エージェントに与えられる操作空間は、モデルが定義する操作空間と正確に一致します。この「不在」こそが主張です。
- **人間をゲートするのと同じ事前条件がエージェントもゲートします。** エージェントが出荷済み注文のキャンセルを試みると、`{ "error": { "code": "SHIPPED_ORDER_CANNOT_BE_CANCELLED", … } }` — 読んで、リカバリして、ユーザーに説明できる機械可読な拒否を受け取ります。

読み取りも同じ方法でスコープされます: すべてのクエリは — 無スコープの管理者ビューであると宣言済みの監査ログを除き — ある actor *として*実行され、モデルに付いた可視性ポリシーが「その actor の世界にどのオブジェクトが含まれるか」を決めます — エージェントのセッションも例外ではありません。

**業務ルールはプロンプトではなく、オントロジーに住む。**

| アプローチ | 読み取り | 書き込み | ルールの担保 |
| --- | --- | --- | --- |
| DB 直結（SQL ツール / DB MCP） | テーブル | 無制限の `UPDATE` | 祈りとプロンプト |
| セマンティックレイヤー / メトリクス MCP | 統制されたメトリクス | — | 対象外（読み取り専用） |
| API ラッパーツール | エンドポイント | エンドポイント単位 | 各バックエンドでバラバラ |
| **Operational Ontology** | オブジェクト・リンク・集計 | **名前つきアクションのみ** | **モデル内の事前条件、監査つき** |

## パターン

概念は 5 つ — オブジェクト・リンク・アクション・edits・監査ログ。データとして定義され、ランタイム（`src/core.ts`）が解釈します:

```ts
const ontology = defineOntology({
  name: 'orders',
  objects: {
    Customer: defineObject({
      primaryKey: 'id',
      properties: { id: z.string(), name: z.string(), region: z.string() },
    }),
    Order: defineObject({
      primaryKey: 'id',
      properties: {
        id: z.string(),
        status: z.enum(['pending', 'shipped', 'cancelled']),
        total: z.number().int(), // minor units — お金は float ではない
        assignee: z.string().nullable(),
      },
      owned: { assignee: null },                       // オントロジー自身の状態、と宣言する
      source: 'north.tbl_order ∪ south.SALES_ORDER',   // 物理データが先にある
    }),
  },
  links: {
    customerOrders: defineLink({ from: 'Customer', to: 'Order', kind: 'one-to-many' }),
  },
  actions: {
    cancelOrder: defineAction({
      object: 'Order',
      targetParam: 'orderId',
      params: { orderId: z.string(), reason: z.string().min(1) },
      preconditions: [
        ({ object }) => object.status === 'shipped'
          ? reject('SHIPPED_ORDER_CANNOT_BE_CANCELLED', `order ${object.id} has already shipped`)
          : undefined,
      ],
      effects: ({ object }) => [modify('Order', object.id, { status: 'cancelled' })],
      writeback: true,
    }),
  },
})
```

定義はただの値です — 列挙でき、diff が取れ、バージョン管理できます。だからこそ MCP の tool 群を導出でき、書き込み経路を閉じられます: 唯一の公開された*操作系*書き込み経路は `Runtime.execute()` で、常に *バリデーション → 事前条件 → 効果 → 編集プランの検証 → 書き戻し → edits + 監査エントリの原子コミット* の順で走ります。（`load()` はソースの再インデックスです — 再生であって判断ではなく、ユーザー API ではなくインフラの境界です。）

読み取りも identity を運びます: すべての `search` / `get` / `traverse` / `aggregate` は `actor` として実行され、オブジェクト型には `visibility` 述語を付けられます — 行レベルセキュリティの最小形で、他のすべてと同じくモデルに住みます。隠されたオブジェクトは存在しないオブジェクトと区別できません。読み取りでも、アクションのターゲットとしても。

edits もまたデータです: `modify`、`create`、`delete` — そして `link` / `unlink`。つまりアクションはノードのプロパティだけでなく、**インスタンスグラフを配線し直せます**。リンクの*型*はモデルの一部であり、ここでは変わりません。どのオブジェクト間にどの辺が実在するかは状態であり、業務状態はアクションを通してしか変わらない — 多重度も含めてです: 1 つの注文に顧客が 2 人つくような `link` はランタイムが拒否します。「この注文を別の顧客に付け替える」は unlink + link + modify であり、他のすべてと同じ事前条件のもとで原子的に適用されます。削除は RESTRICT です: リンクが残っているオブジェクトは死ねません — 先に unlink してください。そしてアクションは*ターゲットレス*にもなれます: 既存のオブジェクトを主語に持たず、プラン全体が生成です。ランタイムはそのようなプランを「オブジェクトを生成し、自分が生成したもの同士を配線する」ことだけに縛ります — 既存の何かに触れる編集は拒否されます。それはターゲットつきアクションの可視性ゲートの内側でやる仕事だからです（[詳細](./IMPLEMENTATION.ja.md#ターゲットレスのスコープ)）。（モデル自体の変更 — 新しいオブジェクト型やリンク型 — はスキーマ進化です。FAQ 参照。）

モデルをクラスではなくデータで持つのは美学の問題ではありません。`class Order { cancel() {} }` は、リフレクション層を上から被せない限り、エージェント用ツールへの列挙も、複数アプリでの共有も、ランタイムからの内省もできません — そしてシグネチャは事前条件について何も語りません。それは 1 つのアプリケーションが私有するドメイン層です。ここでの要点はまさに、ドメイン層が私有物であることをやめる、という点にあります。

## この実装の立ち位置

3 層のうち、このリポジトリが実装するのは中間層だけです。

  <img src="./assets/where-this-sits.svg" alt="3 層 — アプリケーション、Operational Ontology、データ層 — をそれぞれ Foundry での実装とこのリポジトリでの実装に対応づけた図。このリポジトリが実装するのは中間層で、自前のストアを所有する。オントロジーとデータ層の継ぎ目には 2 つの契約: 統合済みの物理データは与えられ、書き戻しは統制された副作用。">
</picture>

**上流との契約:** 統合済みの物理データは*与えられるもの*。パイプライン・データセットのトランザクション・ロールバックはデータ基盤の仕事。

**下流との契約:** 書き戻しは統制された**副作用**であって、分散トランザクションではない（後述）。

クエリ側の層とこのパターンを分ける帰結を 1 つ明記しておきます: **クエリに答えるだけの層は仮想化のままでいられるが、書き込みを受け付ける層は状態を所有しなければならない。** 編集はシステムオブレコードより先に — あるいはその代わりに — ここに存在します（`assignOrder` はどのレガシーにもカラムがない状態を書き込みます — 性質 4 の語彙で言えば ontology-owned な状態です）。だからオントロジーは自前のストアと監査ログを持ちます。一方でこのリポジトリが*所有しない*もの: エンタープライズ規模で読み取りを速くするインデクシング機構（インクリメンタルインデックス、隣接インデックス、検索バックエンド）。それは Foundry が数十億オブジェクトを捌くための手段であり、この層の一実装の事情であって、パターンの一部ではありません。

## 失敗時のセマンティクス

**パターンレベル。** Operational Ontology は、オントロジーとシステムオブレコードの間の整合性*機構*を規定しません — 分散トランザクション、順序契約、outbox、照合ジョブはいずれも実装の選択です。リンク走査に転置インデックスを使うかどうかが実装の選択であるのと同じ枠組みです。ただし 1 つだけ非対称があります: インデックスは契約から観測不能ですが、failure semantics は観測可能 — ユーザーはシステムの乖離を観測できます。だからパターンはここで 1 つだけ要求します: **実装は自らの failure semantics を宣言しなければならない。** 宣言された乖離は工学の問題ですが、本番で発見される乖離はインシデントです。「うちのオントロジー製品は SAP と分散トランザクションを張れる」と言う人がいたら、その宣言を見せてもらってください。

**本実装の宣言。** Foundry の write-back webhook の順序（Foundry が持つ 2 モードのうちの 1 つ — もう一方は編集の*後に* side effect を走らせる）を採用します: `WritebackAdapter` はローカルコミットに先行し、システムオブレコードが拒否すればオントロジーは何も変わりません。逆方向の失敗 — アダプタは成功、ローカルコミットが失敗 — は残ります。[Palantir の webhook ドキュメント](https://www.palantir.com/docs/foundry/action-types/webhooks)も Foundry の write-back モードにおける同じギャップを認めています。それが起きたとき両システムは乖離しており、照合はあなたの仕事です。

この境界を研ぎ澄ますサブ宣言が 3 つあり、詳細は[実装ノート](./IMPLEMENTATION.ja.md#失敗時のセマンティクスの詳細)にあります: 不正なものはシステムオブレコードに決して届かない（アダプタが走る前に、編集プラン全体がコミット自身のコードでドライランされる）。監査ログはどちら向きの失敗でもプラン全体を記録し、照合の材料になる。そして「すべての試行は監査される」が誠実に意味するのは、*このランタイムが完了まで観測できた*すべての試行である。

自分のストアの内側では、ランタイムは誠実にトランザクショナルです: アクションの edits と監査エントリは原子的にコミットされ（単一の SQLite トランザクション）、拒否された試行も記録されます。

**事前条件と鮮度。** 事前条件はオントロジーストア — 最後にインデックスされたスナップショット + 適用済みの編集 — に対して評価されます。ランタイムの write-back ステップ自体はソース側で不変条件を再検証しません: ソースシステムがオントロジーの背後で変わりうるなら、不変条件が成立するのはオントロジーの見ている世界に対してです。このギャップを狭めるのはアダプタの選択 — 条件付き write-back、compare-and-set、システムオブレコード側での再検証 — であり、デモのアダプタはその選択をしています: ガード付き `UPDATE` により、ERP が古くなったキャンセルを自分で拒否できる。

**並行編集。** 同時に実行されるアクションがどう合成されるかは implementation-defined — ロック、バージョン、楽観的並行性制御 — であり、他のすべてと同じく宣言義務です。本実装は同期的なシングルライター: アクションはランタイムにより直列化され、一度に 1 つずつ実行されます。`WritebackAdapter` インターフェースが同期なのも同じ理由です。本物のネットワーク越し write-back はこの直列化を壊すため、そこへ進む実装は「何がそれを置き換えるか」を宣言しなければなりません。ランタイムは自分のデータベーストランザクションも所有します — 呼び出し側がアクションをトランザクションで包むことも、ルールやアダプタがオントロジーストアにトランザクションを開くこともできません。その強制の仕組みは[実装ノート](./IMPLEMENTATION.ja.md#トランザクションの所有権)にあります。

**再インデックス × 編集。** オントロジーストアは 2 種類の状態を持ちます: ソースからインデックスされたベースと、その上にアクションが加えた編集。ソースシステムは動き続けるので、このパターンの実装者は必ず「再インデックスが編集に出会ったとき何が起きるか」を決めなければなりません。Foundry は編集を独立したレイヤーに保持し、再インデックスされたベースの上に再適用します。この実装も同じことを、最小化して、authority 宣言に駆動させて行います。スナップショットが語れるのは source-backed な状態だけで、ontology-owned な状態は生き延びます — オーバーレイから再適用されるか、そもそもスナップショットの手が届かないままか。そして ontology-owned な編集を孤児にする再インデックスは全体が拒否され、以前の状態が立ったまま残ります: それは照合上の判断であり、ランタイムは照合上の判断を黙って下しません。完全な規則 — 削除が生き残った編集に出会うとどうなるか、部分スナップショットはどうなるか — は[実装ノート](./IMPLEMENTATION.ja.md#再インデックスと編集)にあります。

## Non-goals（作らないもの）

スコープは v0 として、設計上凍結されています。これは定義に付属するリファレンス実装であり、成長を拒むことで可読性を保ちます:

- **UI ビルダーは作らない。** アプリケーションはオントロジーの消費者であって、オントロジーの一部ではない。
- **パイプラインフレームワークは作らない。** 統合は前提条件。デモは生 SQL で済ませる。
- **インデクシング基盤は作らない。** デモスケールでは素朴なクエリで足りる。スケールは実装の性質であって、パターンの性質ではない。v0 では結果セットは有界ではない。
- **1 つのオントロジー = 1 つの bounded context。** フェデレーション — モデルが複数あるとき誰が所有するのか — は現実の問題だが、スキーマ進化と同じく v0 では扱わない。
- **リンク属性・複合主キーは（まだ）ない。** デモの注文明細の数量は意図的にデータ層に留めている。第一級の `OrderLine` にするかリンクの属性にするかは v0.2 の決定事項。
- **OWL/RDF 対応はしない。** 学術オントロジーはセマンティックのみ — アクションもキネティクスもない。別の仕事のための別の道具。
- **汎用の認可システムは作らない。** パターンレベルの部分はここにある — identity は全呼び出しを流れ（宣言済みの例外は監査ログの管理者ビュー）、可視性はモデルに付く。機構（グループ、属性、ポリシー言語、セルレベルセキュリティ、伝播）はポリシーエンジンの仕事。
- **npm パッケージにしない。** フォークして使う。依存はしない。
- **Foundry の代替ではない。** Foundry は 3 層すべてを垂直統合したこのパターンの一実装。このリポジトリは中間層に名前を与え、実演する。

## Prior art（先行事例）

- **Palantir Foundry Ontology** — このパターンのリバースエンジニアリング元。[semantic/kinetic の語彙](https://www.palantir.com/docs/foundry/ontology/overview)、[action types](https://www.palantir.com/docs/foundry/action-types/overview)、[write-back webhooks](https://www.palantir.com/docs/foundry/action-types/webhooks)、そして Ontology MCP。Palantir がエージェント向けオントロジー接続面を出荷している事実は、このカテゴリが実在する証拠であって、彼らの専有物である証明ではない。
- **DDD、CQRS、イベントソーシング** — 部品は意図的に古い。エンティティ、集約、コマンド、ガードつき状態遷移、追記型ログ。新しいのは配置: ドメイン層を単一アプリケーションから引き上げ、*他システムの*データの上に置き、多数のアプリとエージェントで共有する。
- **セマンティックレイヤー**（dbt、Cube、AtScale 等）と**ナレッジグラフ / OWL / RDF** — 統制された読み取りはあるが、統制された書き込みはない。このパターンの対比先となる隣接カテゴリ。
- **"Operational ontology"** という語は学術的なオントロジー工学で別の意味の散発的な先行使用がある。このリポジトリでは冒頭で定義した意味に限定して使う。

## FAQ

**これはバリデーション付き CRUD では？** 部品は馴染み深いもの。しかし構成が違います。CRUD のバリデーションは 1 つのアプリの内部、そのアプリが所有するテーブルの上に住む。ここではモデルが*他システムの所有する*データの上に座り、すべての消費者（UI・スクリプト・エージェント）に共有され、アクション以外の書き込み経路を閉じ、全試行を監査し、受理した変更をシステムオブレコードへ書き戻す。近いのは「アプリケーションから取り出され、他人のデータの上に置かれた CQRS のコマンド層」です。

**ナレッジグラフだって書けるのでは？** 書けます — SPARQL UPDATE は `status = 'cancelled'` に喜んで書き換えてくれますし、`WHERE` 句や SHACL のシェイプを使えば条件付きにもできます。スタックが渡してくれないのは、契約の残り全部を 1 つの第一級ユニットとして扱うこと: 名前のついた業務操作、機械可読な拒否、試行の監査証跡、システムオブレコードへの書き戻し。違いは能力ではありません — トリプルストアの上にこれを全部組むことはできる — モデルから何が名前付き・統制付き・第一級で出てくるか、です。

**なぜ OWL/RDF ではないのか？** あれはモノが*何であるか*（セマンティック）のモデル。このパターンの差別化の半分は*何ができるか*（キネティック）: アクション、事前条件、監査、書き戻し。推論器は注文をキャンセルできない。

**なぜ YAML ではなく TypeScript の定義なのか？** 業務ルールはコードだから — YAML のルール式言語はおもちゃのルールエンジンが生まれる経路そのもの。TS のオブジェクトリテラルなら、モデルの列挙可能性を保ちつつ、ルールを普通の型付きコードとして書ける — 型が守るのは言語に対してであって、モデル自身のスキーマに対してではない（[ステータス](#ステータス)参照）。構造はデータ、ルールは関数 — Foundry が Ontology Manager と Functions の間で行っているのと同じ分割。

**トランザクションとロールバックはどうなっている？** 3 つの異なるドメインに 3 つの異なる答えがあります: データセットの版管理・ロールバックはデータ層の仕事（Foundry ではカタログのトランザクションとブランチ）。アクションの edits の原子適用はこの層の仕事（ここでは実際の SQLite トランザクションで実装）。そしてシステム横断の write-back の整合性機構は implementation-defined — パターンが要求するのは*宣言*であって仮定ではない。本実装は write-back 先行の順序を宣言している。[失敗時のセマンティクス](#失敗時のセマンティクス)参照。

**権限・セキュリティはどうなっている？** この質問には 3 つの別物が隠れていて、他のすべてと同じ線で切り分けられます。*認証*はパターンの外: identity は確立済みで渡されてくる（ここでは `actor` は自己申告の文字列 — この実装が実証するのは*配置*であって保護ではない）。*認可*の**配置**はパターンの一部 — Foundry は dynamic security をオントロジーのキネティック要素に数え、ポリシーはオブジェクト型とアクションに付いて、全消費者の読み書き両方を拘束する — 一方で**機構**（グループ、属性、ポリシー言語、セルレベルセキュリティ、データ層からの伝播）は implementation-defined。そして*事前条件*はどちらでもない: あれは妥当性であって権限ではない。保持すべき語彙:

- **visibility** — あなたには見えない
- **permission** — あなたには不可（*not you*）
- **precondition** — 誰にも不可能（*not ever*）

エージェントのリカバリはそれぞれ異なるため、本実装はこの 3 つを区別して保つ。Foundry は submission criteria の中で permission と validity を混ぜているが、それでも適合する — 分離は推奨であって要求ではない。

宣言済みの設計判断が 2 つ続きます。第一にスロット: `preconditions` は必須キーで、空リストは表明された決定（ゲート付き書き込みはパターンの核だから）。`visibility` は optional キー（認可の*存在*は implementation-defined だから）。第二に極性 — failure semantics と同じく、デフォルトは宣言義務: 本実装は **fail-open**（`visibility` 不在 = 全員に可視）。認証を持たない参照実装は意味のある fail-closed になれず、装えばセキュリティ劇場になる。Foundry のベースラインは正反対 — discretionary な grant がゼロからアクセスを広げ、mandatory な marking がその上から連言的に拒否する。fail-closed 配備は `visibility` スロットを optional から必須に反転させることから始まります — が、そこでは終わりません: その下に本物の認証、アクション権限、スコープされた監査アクセスが必要です。宣言済みの面をもう 1 つ: 監査ログの読み取り API は無スコープです — visibility フィルタが適用されない管理者向けビューで、他と同じく fail-open。

**Foundry の Functions と派生プロパティは？** Foundry はキネティック要素を 3 つ数えます: actions、functions、dynamic security。function-*backed* な action はこのパターンの内側に既にいます — 事前条件と効果は変更を*記述する*普通のコードです: effects は編集プランを返すだけで自分では何も実行せず、副作用はアダプタの仕事。同じ分割（構造はデータ、ルールは関数）です。読み取り時計算 — 派生プロパティ、クエリ関数 — は意図的に外側です: このパターンの差別化の半分は統制された*書き込み*であって、計算された*読み取り*ではない。dynamic security は上の権限の話です。

**エージェントがリトライしたら？** 冪等性は implementation-defined です — そして宣言する価値があります。エージェントはリトライするからです。本実装に冪等キーはありません: リトライされた `cancelOrder` は自分の事前条件（`ORDER_ALREADY_CANCELLED`）に拒否されます — ルールの働きによる自然な冪等性であって、保証ではない — そして write-back と交錯するリトライは、ソース側で副作用を二重適用しえます。アクションが自然に冪等でないなら、params に invocation id を入れて — 他と同じく監査に載せる — のが最小の出発点です: 得られるのは相関であり、実際の重複排除にはその id への一意性チェックが必要です。

**オントロジー自体はどう変化する？** スキーマ進化 — 新しいオブジェクト型、プロパティの変更、リンクの廃止 — は現実の問題ですが、bounded context と同じくここではスコープ外です: Foundry にはバージョニングとプロポーザルの機構があり、学術界は *ontology evolution* として研究しています。このリポジトリが提供するのは、安全に進化するための前提条件です: モデルはただの値なので、他のコードと同じように diff が取れ、バージョン管理でき、レビューできます。リバースドメインモデリングは一回きりの工程ではなく、モデルはビジネスに合わせて再適合され続けます。

**フロントエンドは自前で用意する？** はい。アプリケーションとの契約は 2 動詞 — クエリ（`search`/`get`/`traverse`/`aggregate`）と `execute(action)` — で、人間もエージェントも同じ、そしてすべての呼び出しはある actor *として*行われます。ダッシュボードは 1 つ目の動詞、「キャンセル」ボタンは 2 つ目。ルールはフロントエンドではなくモデルに従います。
