# Operational Ontology

> **Operational Ontology（操作できるオントロジー）とは、自分が所有していないシステム群のデータの上に構築される共有ドメインモデルである — オブジェクト・リンク・アクションで構成され、読み取りはモデルを辿り、書き込みは業務ルールを持つアクションだけを通り、監査され、システムオブレコードへ書き戻される。**
>
> セマンティックレイヤーはビジネスを「読む」ためのもの。Operational Ontology はビジネスを「動かす」ためのもの。

Palantir Foundry の Ontology はこのパターンの一実装です。このリポジトリはもう一つの実装 — 最小で可読なリファレンス実装、つまり概念の最小化です。これはフレームワークではなく「動く定義」です。読んで、フォークして、アイデアを盗んでください。

## 4 つの性質

以下の 4 つがすべて成り立つとき、そのシステムはこのパターンを実装しています:

1. **セマンティックなオブジェクトとリンク。** 業務のエンティティとその関係が、先に存在していた物理データの上に明示的にモデリングされている。
2. **アクションに一本化された書き込み。** 状態は名前のついたアクションを通してしか変化しない。汎用の更新経路は存在しない。
3. **アクションに紐づく業務ルール。** 事前条件はドメインの不変条件（「出荷済みの注文はキャンセルできない」）であり、機械可読なエラーで拒否される — アクセス制御でも UI バリデーションでもない。
4. **システムオブレコードへの書き戻し（write-back）。** 変更は統制された順序つきの副作用として基幹システムへ伝播する。真実の所在は元の場所のまま。

リトマス質問: **「あなたのセマンティックレイヤーから注文をキャンセルできますか？」** 答えが No なら、それは読み取り層です — 有用ですが、別のものです。

## クイックスタート

```sh
pnpm install
pnpm demo    # 物理データ → 統合 → インデックス → 読み取り → 書き込み → 拒否 → 書き戻し
pnpm test    # 仕様のテスト化
```

デモの題材は、このリポジトリが付属する記事と同じシナリオです: ある企業が競合を買収し、**スキーマもステータスのエンコーディングも異なる 2 つのレガシー受注システム**を抱え込む。数十行の生 SQL で統合し、その上に `Customer` / `Order` / `Product` をオントロジーとしてモデリングする。そして:

- リンク走査が「この製品を含む注文はどれか？」に両システム横断で答える
- `assignOrder` は**どのレガシーシステムにも存在しない状態**を書き込む — 編集はソースの上の層に住める
- 出荷済み注文への `cancelOrder` は `SHIPPED_ORDER_CANNOT_BE_CANCELLED` で**拒否される**
- 未出荷注文への `cancelOrder` は成功し、レガシー ERP の行が**実際に書き変わる**
- すべての試行が — 適用も拒否も — 監査ログに残る

## AI エージェント向け（MCP）

```sh
pnpm mcp     # 同じオントロジーを stdio 経由でエージェントに公開
```

MCP のツール群は*モデルから生成*されます: `search_order`、`traverse_customer_orders`、`cancel_order`、`read_audit_log`…… クエリの形とアクションごとに 1 ツール。注目すべき点が 2 つあります:

- **raw SQL ツールは意図的に存在しません。** エージェントに与えられる操作空間は、モデルが定義する操作空間と正確に一致します。この「不在」こそが主張です。
- **人間をゲートするのと同じ事前条件がエージェントもゲートします。** エージェントが出荷済み注文のキャンセルを試みると、`{ "code": "SHIPPED_ORDER_CANNOT_BE_CANCELLED", ... }` — 読んで、リカバリして、ユーザーに説明できる機械可読な拒否を受け取ります。

**業務ルールはプロンプトではなく、オントロジーに住む。**

| アプローチ | 読み取り | 書き込み | ルールの担保 |
| --- | --- | --- | --- |
| DB 直結（SQL ツール / DB MCP） | テーブル | 無制限の `UPDATE` | 祈りとプロンプト |
| セマンティックレイヤー / メトリクス MCP | 統制されたメトリクス | — | 対象外（読み取り専用） |
| API ラッパーツール | エンドポイント | エンドポイント単位 | 各バックエンドでバラバラ |
| **Operational Ontology** | オブジェクト・リンク・集計 | **名前つきアクションのみ** | **モデル内の事前条件、監査つき** |

## パターン

概念は 5 つ。データとして定義され、ランタイム（`src/core.ts`）が解釈します:

```ts
const ontology = defineOntology({
  objects: {
    Order: defineObject({
      primaryKey: 'id',
      properties: { id: z.string(), status: z.enum(['pending', 'shipped', 'cancelled']), total: z.number() },
      source: 'north.tbl_order ∪ south.SALES_ORDER',   // 物理データが先にある
    }),
  },
  links: {
    customerOrders: defineLink({ from: 'Customer', to: 'Order', kind: 'one-to-many' }),
  },
  actions: {
    cancelOrder: defineAction({
      object: 'Order',
      targetParam: 'orderId',
      params: { orderId: z.string(), reason: z.string().min(1) },
      preconditions: [
        ({ object }) => object.status === 'shipped'
          ? reject('SHIPPED_ORDER_CANNOT_BE_CANCELLED', `order ${object.id} has already shipped`)
          : undefined,
      ],
      effects: ({ object }) => [modify('Order', object.id, { status: 'cancelled' })],
      writeback: true,
    }),
  },
})
```

定義はただの値です — 列挙でき、シリアライズでき、diff が取れます。だからこそ MCP の tool 群を導出でき、書き込み経路を閉じられます: 状態を変更する公開メソッドは `Runtime.execute()` ただ一つで、常に *バリデーション → 事前条件 → 効果 → 書き戻し → edits + 監査エントリの原子コミット* の順で走ります。

モデルをクラスではなくデータで持つのは美学の問題ではありません。`class Order { cancel() {} }` はエージェント用ツールに列挙できず、複数アプリで共有できず、ランタイムから内省できません — それは 1 つのアプリケーションが私有するドメイン層です。ここでの要点はまさに、ドメイン層が私有物であることをやめる、という点にあります。

## この実装の立ち位置

3 層のうち、このリポジトリが実装するのは中間層だけです。

```
┌───────────────────────────────────────────────────────────┐
│  アプリケーション層    Foundry では Workshop / OSDK          │
│                       → ここでは CLI デモ + MCP エージェント  │
├───────────────────────────────────────────────────────────┤
│  OPERATIONAL ONTOLOGY オブジェクト・リンク・アクション・監査   │  ← このリポジトリ
│                       自前のストアを所有する（編集はここに住む）│
├───────────────────────────────────────────────────────────┤
│  データ層             パイプライン、データセットの版管理、      │
│                       インデクシング基盤（Foundry では        │
│                       Funnel/OSv2）→ ここでは fixture + 生SQL │
└───────────────────────────────────────────────────────────┘
```

**上流との契約:** 統合済みの物理データは*与えられるもの*。パイプライン・データセットのトランザクション・ロールバックはデータ基盤の仕事。

**下流との契約:** 書き戻しは統制された**副作用**であって、分散トランザクションではない（後述）。

読み取り専用の層との違いを分ける帰結を 1 つ明記しておきます: **読み取り専用の層は仮想化のままでいられるが、書き込みを受け付ける層は状態を所有しなければならない。** 編集はシステムオブレコードより先に — あるいはその代わりに — ここに存在します（`assignOrder` はどのレガシーにもカラムがない状態を書き込みます）。だからオントロジーは自前のストアと監査ログを持ちます。一方でこのリポジトリが*所有しない*もの: エンタープライズ規模で読み取りを速くするインデクシング機構（インクリメンタルインデックス、隣接インデックス、検索バックエンド）。それは Foundry が数十億オブジェクトを捌くための手段であり、この層の一実装の事情であって、パターンの一部ではありません。

## 失敗時のセマンティクス

正直さのためのセクションです。システム横断の整合性はこのパターンで最も難しい未解決部分であり、このリポジトリは解いていません — 特筆すべきことに、参照元のベンダーも解いていません。Foundry の write-back webhook はオントロジー編集より*先に*実行され、失敗時は編集をブロックしますが、逆方向の失敗（外部システムは成功、オントロジー編集が失敗）は起こり得ると Palantir 自身のドキュメントに明記されています。この実装はその契約を正確に写しています: `WritebackAdapter` はローカルコミットの前に走り、throw すればローカルは何も変わらない。アダプタ成功後にローカルコミットが失敗すればシステムは乖離し、照合はあなたの仕事です。「うちのオントロジー製品は SAP と分散トランザクションを張れる」と言う人がいたら、もっと厳しく質問してください。

自分のストアの内側では、ランタイムは誠実にトランザクショナルです: アクションの edits と監査エントリは原子的にコミットされ（単一の SQLite トランザクション）、拒否された試行も記録されます。

## Non-goals（作らないもの）

スコープは設計として凍結されています。これは定義に付属するリファレンス実装であり、成長を拒むことで可読性を保ちます:

- **UI ビルダーは作らない。** アプリケーションはオントロジーの消費者であって、オントロジーの一部ではない。
- **パイプラインフレームワークは作らない。** 統合は前提条件。デモは生 SQL で済ませる。
- **インデクシング基盤は作らない。** デモスケールでは素朴なクエリで足りる。スケールは実装の性質であって、パターンの性質ではない。
- **OWL/RDF 対応はしない。** 学術オントロジーはセマンティックのみ — アクションもキネティクスもない。別の仕事のための別の道具。
- **汎用の認可システムは作らない。** 事前条件は操作が*妥当か*を決める。誰が*試行できるか*はポリシーエンジンの仕事。
- **npm パッケージにしない。** フォークして使う。依存はしない。
- **Foundry の代替ではない。** Foundry は 3 層すべてを垂直統合したこのパターンの一実装。このリポジトリは中間層に名前を与え、実演する。

## Prior art（先行事例）

- **Palantir Foundry Ontology** — このパターンのリバースエンジニアリング元。[semantic/kinetic の語彙](https://www.palantir.com/docs/foundry/ontology/overview)、[action types](https://www.palantir.com/docs/foundry/action-types/overview)、[write-back webhooks](https://www.palantir.com/docs/foundry/action-types/webhooks)、そして Ontology MCP。Palantir がエージェント向けオントロジー接続面を出荷している事実は、このカテゴリが実在する証拠であって、彼らの専有物である証明ではない。
- **DDD、CQRS、イベントソーシング** — 部品は意図的に古い。エンティティ、集約、コマンド、ガードつき状態遷移、追記型ログ。新しいのは配置: ドメイン層を単一アプリケーションから引き上げ、*他システムの*データの上に置き、多数のアプリとエージェントで共有する。
- **セマンティックレイヤー**（dbt、Cube、AtScale 等）と**ナレッジグラフ / OWL / RDF** — 統制された読み取りはあるが、統制された書き込みはない。このパターンの対比先となる隣接カテゴリ。
- **"Operational ontology"** という語は学術的なオントロジー工学で別の意味の散発的な先行使用がある。このリポジトリでは冒頭で定義した意味に限定して使う。

## FAQ

**これはバリデーション付き CRUD では？** 部品は馴染み深いもの。しかし構成が違います。CRUD のバリデーションは 1 つのアプリの内部、そのアプリが所有するテーブルの上に住む。ここではモデルが*他システムの所有する*データの上に座り、すべての消費者（UI・スクリプト・エージェント）に共有され、アクション以外の書き込み経路を閉じ、全試行を監査し、受理した変更をシステムオブレコードへ書き戻す。近いのは「アプリケーションから取り出され、他人のデータの上に置かれた CQRS のコマンド層」です。

**なぜ OWL/RDF ではないのか？** あれはモノが*何であるか*（セマンティック）のモデル。このパターンの差別化の半分は*何ができるか*（キネティック）: アクション、事前条件、監査、書き戻し。推論器は注文をキャンセルできない。

**なぜ YAML ではなく TypeScript の定義なのか？** 業務ルールはコードだから — YAML のルール式言語はおもちゃのルールエンジンが生まれる経路そのもの。TS のオブジェクトリテラルなら、モデルの列挙可能性*と*ルールの型チェックを両立できる。構造はデータ、ルールは関数 — Foundry が Ontology Manager と Functions の間で行っているのと同じ分割。

**トランザクションとロールバックはどうなっている？** 3 つの異なるドメインに 3 つの異なる答えがあります: データセットの版管理・ロールバックはデータ層の仕事（Foundry ではカタログのトランザクションとブランチ）。アクションの edits の原子適用はこの層の仕事（ここでは実際の SQLite トランザクションで実装）。システム横断の write-back は、我々の知る限り*どの*実装でもトランザクショナルではない — [失敗時のセマンティクス](#失敗時のセマンティクス)参照。

**フロントエンドは自前で用意する？** はい。アプリケーションとの契約は 2 動詞 — クエリ（`search`/`get`/`traverse`/`aggregate`）と `execute(action)` — で、人間もエージェントも同じです。ダッシュボードは 1 つ目の動詞、「キャンセル」ボタンは 2 つ目。ルールはフロントエンドではなくモデルに従います。
