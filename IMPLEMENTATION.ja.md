# 実装ノート

[README](./README.ja.md) はパターンを定義し、本リポジトリの宣言済みの答えを要約しています。この文書はその工学的な付録です: 答えの背後にある正確な機構 — エラーコード、エッジケース、それらが積み上げる保証。ここにあるものはパターンではありません。すべては一実装が宣言する内容であり、記述された振る舞いはすべて [`tests/core.test.ts`](./tests/core.test.ts) のテストで固定されています。

ここで名前を挙げる拒否はすべて機械可読です: 該当したアクションは `{ ok: false, error: { code, message } }` を返し、試行は監査ログに残ります。

## Authority の検査

モデルは、オントロジーが所有する状態（オブジェクト型とリンクの `owned`）と、source-backed な変更（アクションの `writeback: true`）を宣言します。ランタイムはすべての編集プランを `owned` 宣言に照らして分類し、アクションの宣言と矛盾するプランを拒否します:

- **`UNDECLARED_SOURCE_WRITE`** — プランは source-backed な状態を変えるのに、アクションが `writeback: true` を宣言していない。故郷に帰らないソース真実へのローカルな変更は、性質 4 が禁じる影のコピーそのものである。
- **`MISDECLARED_WRITEBACK`** — アクションは `writeback: true` を宣言しているのに、プランは ontology-owned な状態しか変えない。ソースに属するものが何もない。
- **`MIXED_AUTHORITY`** — プランが両方の状態を変えている。1 つの編集の内側でも、編集をまたいでも。本実装はプランを丸ごとルーティングするので、アクションは境界線のどちらか片側に立たなければならない — またぐなら分割する。（edit 単位のルーティングは次版の課題。）
- **`SOURCE_CREATE_UNSUPPORTED`** — プランが source-backed な型のオブジェクトを生成しようとしている。*ソース側で*行を生成することは現実に存在する — write-back は運べる — が、本実装は実証していないため、中途半端に支えるのではなく拒否する。生成は ontology-owned な型に限られる。（これも次版の課題。）

空のプランは境界線のどちら側にも触れません: アダプタは呼ばれず、監査エントリだけがコミットされます。write-back を宣言したアクションにアダプタが構成されていなければ、拒否されます（**`NO_WRITEBACK_ADAPTER`**）。

## 失敗時のセマンティクスの詳細

README は write-back 先行の順序を宣言しています: アダプタはローカルコミットに先行し、システムオブレコードが拒否すればこちらは何も変わらない — そして逆方向の失敗（アダプタは成功、ローカルコミットが失敗）は残る。この境界を 3 つの機構が研ぎ澄まします:

**不正なものは境界を越えない。** アダプタが走る前に、編集プラン全体が「必ずロールバックされるトランザクション」の中で適用されます — コミット自身のコードによる、コミットのドライラン。同期がずれうる第二のバリデータではありません。オントロジーストアが挙げうるすべての違反（スキーマ、多重度、リンクの端点）は、何かがシステムオブレコードに届く*前に*拒否されます。残る乖離は宣言済みの逆方向失敗だけです。アダプタはルーティングに必要な材料も前もって受け取ります: 検証済みプランと、ランタイムがロードしたターゲットオブジェクト（`meta.target`）の自分用コピーです。オントロジーストアを読む必要はありません。

**監査ログが照合の材料を運ぶ。** **`WRITEBACK_FAILED`** の拒否はアダプタが見たプラン全体を記録します — アダプタは throw する前にプランを部分適用したかもしれません。ソース側の原子性はアダプタの契約であって、このランタイムの契約ではない。逆方向の失敗は **`COMMIT_FAILED`** として、プランを添えて監査されます: write-back 先行のアクションの後では、その edits こそが既にソースへ届いたものだからです。

**「すべての試行は監査される」の誠実な限界。** それは*このランタイムが完了まで観測できた*すべての試行を意味します。ソース更新とローカルコミットの間でプロセスが死ねば、編集も監査エントリも失われます。この窓を閉じるには永続化された pending-invocation 記録が必要で、本実装は持っていません。

書き込み経路の内側のクラッシュも監査されます。コードが場所を語ります: ストレージ障害は **`READ_FAILED`**、モデルコード — 可視性述語、事前条件、効果関数 — が throw したなら **`RULE_CRASHED`**。その後、エラーは呼び出し側に浮上します。

監査の書き込み自体が失敗の原因になってはなりません。JSON の round trip を生き延びない params はモデルが走る前に **`INVALID_PARAMS`** で拒否され、ログがエンコードできないものは `$unserializable` プレースホルダとして記録されます — 欠損した監査エントリより、劣化した監査エントリの方がましだからです。

## トランザクションの所有権

強制されるルールは 1 つです: **呼び出し側はランタイムを包めない。** `execute()` と `load()` は、呼び出し側が開いたトランザクションの中では実行を拒否します。その中では「コミット済み」は実際には「呼び出し側が savepoint をロールバックするまで」を意味し、適用・監査済みのアクションが成功報告の後に静かに巻き戻せてしまうからです。これは原子性の保証であって、侵入防御ではありません。

境界の残りは**防御ではなく宣言**です。このランタイムはプロセス内ライブラリです: データベースハンドルを持つコード — 呼び出し側、ルール、write-back アダプタ — は、直接の `UPDATE` でアクションゲートを物理的に迂回でき、プロセス内の検査でそれを防ぐことはできません。だから境界は契約です: ルールとアダプタはオントロジーストアに触れてはならない。（アダプタに触れる理由はありません — 編集プランとターゲットオブジェクトの自分用コピーを受け取り、話す相手はシステムオブレコードだけです。）以前の版はこの契約の観測可能な一部 — ストアに開いたまま残されたトランザクション — を検知していましたが、v0.2 で削除しました: 侵入のうち一つの形しか捕まえられず、最も単純な形（autocommit の直接 UPDATE）を素通しする検査は、防御のように見えて防御ではないからです。本物の境界が必要な配備は、ランタイムをプロセス境界の向こうに置きます — MCP サーバーがまさにその形です。

## ストアの境界

ストアは JSON を保持します。したがって保存されるすべての値は、JSON の round trip を無変更で生き延びなければなりません。クラスインスタンス、`Date`、`NaN`、`Map`、どの深さであれ `undefined` — いずれも変わった姿で戻るか脱落するため、それを含む行はどの入口から来ても（アクションでも `load()` でも）書き込みのたびに拒否されます。`owned` プロパティの宣言済みデフォルトは定義時に、アクションの params は入口で、同じ検査を受けます（上の監査の項参照）。

1 つの義務は検査ではなく宣言です: **プロパティスキーマは検証せよ、変換するな。** ランタイムは保存された値を後の書き込みで同じスキーマに再び通すので、変換するスキーマ（`z.coerce.date()`、`.transform(…)`）は次のパスで自分の出力を拒否するか、静かに書き換えます。以前の版はこれを書き込みごとの不動点検査で強制していましたが、v0.2 はモデル作者の契約として明文化する方を選びました。

## 再インデックスと編集

スナップショットのセマンティクスは、ロードされた型ごとに: ベースを置き換え、編集レイヤーを再適用する。詳細は:

- **スナップショットが語れるのは source-backed な状態だけ。** ontology-owned なプロパティを供給する行は拒否されます。ontology-owned な型の行、ontology-owned なリンクのインスタンスも同様です — ソースはそのどれにも権限を持ちません。
- **ontology-owned なプロパティはオーバーレイ経由で生き延びる。** それらへの編集は (type, pk) をキーとするオーバーレイに住み、再インデックス後、現在のパッチが新しいベースの上に再適用されます。宣言済みデフォルトに戻された編集はオーバーレイから刈り取られます — 編集のクリアは生存義務のクリアでもあります。（比較は構造的で、キー順序で「デフォルトに戻った」を偽装することも隠すこともできません。）
- **ontology-owned な型とリンクはその場で生き延びる。** `load()` は触れることを拒否するので、オーバーレイは不要です。
- **競合はロード全体を拒否し**、以前の状態が立ったまま残ります。ontology-owned な編集を抱えたままベース行が消えるのは照合上の判断であり、ランタイムは照合上の判断を黙って下しません: 編集をクリアするか行を復元してから、再ロードしてください。編集のクリアもまたアクションなので、照合すら書き込みゲートの内側に留まります。
- **プロパティの所有をやめたモデルは、同じ競合のスキーマ進化版。** モデルがもう ontology-owned と宣言していないキーを運ぶオーバーレイパッチはロードを拒否します — その状態の運命を決めるのはリフレッシュではなく、明示的なスキーマ進化の仕事だからです。
- **部分スナップショットも同じ帰結。** ロードされない型に生き残った状態がモデルの制約に違反するなら、再インデックス全体が拒否されロールバックされます。
