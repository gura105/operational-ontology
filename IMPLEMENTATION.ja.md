# 実装ノート

[README](./README.ja.md) はパターンを定義し、本リポジトリが宣言した答えを要約しています。この文書は、その答えの背後にある機構、つまり処理順序・エラーコード・エッジケースを説明します。ここに書かれていることはパターンそのものではなく、すべてこの一実装が宣言する内容です。記述された振る舞いはすべて [`tests/core.test.ts`](./tests/core.test.ts) のテストで固定されています。

この文書に登場する拒否は、すべて機械可読です。拒否されたアクションは `{ ok: false, error: { code, message } }` を返し、その試行は監査ログに残ります。

## Authority の検査

モデルは所有を 2 箇所で宣言します。オブジェクト型とリンクの `owned` が ontology-owned な状態に印を付け、アクションの `writeback: true` がその変更を source-backed と宣言します。ランタイムはすべての編集プランを `owned` 宣言に照らして分類し、アクションの宣言と矛盾するプランを拒否します。

| プランの内容 | アクションの `writeback` | 結果 |
| --- | --- | --- |
| source-backed な状態を変える | なし | 拒否：**`UNDECLARED_SOURCE_WRITE`** |
| ontology-owned な状態しか変えない | あり | 拒否：**`MISDECLARED_WRITEBACK`** |
| 両方の種類を変える（1 つの編集内でも、編集をまたいでも） | どちらでも | 拒否：**`MIXED_AUTHORITY`** |
| source-backed な型のオブジェクトを生成する | どちらでも | 拒否：**`SOURCE_CREATE_UNSUPPORTED`** |

各行の理由は次のとおりです。宣言なきソース書き込みは、ソース所有のデータをローカルで変えながらソースに届けない構成であり、性質 4 が禁じるものそのものです。宣言と裏腹の write-back は、プランの中にソースへ属するものが 1 つもありません。混在プランを拒否するのは、本実装がプランを丸ごとルーティングするためで、両方が必要ならアクションを分割してください（edit 単位のルーティングは今後の課題です）。ソース側での行の生成は現実にあり得ますし write-back で運ぶことも可能なはずですが、本実装では実証していないため、中途半端に対応せず拒否します。生成は ontology-owned な型に限られます（これも今後の課題です）。

空のプランは境界のどちら側にも触れません。アダプタは呼ばれず、監査エントリだけがコミットされます。write-back を宣言したアクションにアダプタが構成されていない場合は、**`NO_WRITEBACK_ADAPTER`** で拒否されます。

検査の順序は、妥当性が先、authority が後です。プラン全体がまずコミットと同じコードでドライランされるため、ストアが受け付けないプランは、たとえ authority の境界もまたいでいたとしても **`INVALID_EDITS`** になります。

宣言された 4 つの答えそのものも、`Runtime.declarations` として実行時に列挙でき、テストで固定されています。

## 失敗時のセマンティクスの詳細

宣言している順序は write-back 先行です。アダプタはローカルコミットより先に走り、システムオブレコードが拒否すればオントロジー側は何も変わりません。残るリスクは逆向きの失敗、つまりアダプタは成功したのにローカルコミットが失敗するケースで、これが起きると両システムは乖離します。このリスクを 3 つの仕組みが狭めています。

**不正なデータは境界を越えません。** アダプタが走る前に、編集プラン全体が「必ずロールバックされるトランザクション」の中で一度適用されます。コミットと同じコードによるドライランであり、実装がずれていく可能性のある第二のバリデータではありません。ストアが検出できる違反（スキーマ、多重度、リンクの端点）はすべて、何かがシステムオブレコードに届く前に拒否されます。アダプタは必要な入力を前もって、それぞれ自分専用のコピーとして受け取ります。検証済みのプランと、ランタイムがロードしたターゲットオブジェクト（`meta.target`）です。そのため、アダプタがオントロジーストアを読む必要はありません。

**監査ログは、どちら向きの失敗も記録します。** **`WRITEBACK_FAILED`** の拒否は、アダプタが見たプラン全体を記録します。ソース側の原子性はアダプタの契約でありこのランタイムの契約ではないため、アダプタは throw する前にプランを部分的に適用してしまっているかもしれないからです。逆向きの失敗は **`COMMIT_FAILED`** として、やはりプランを添えて監査されます。write-back 先行のアクションの後では、その編集こそが既にソースへ届いた内容だからです。どちらのエントリも照合の材料になります。

**「すべての試行は監査される」には、明示された限界があります。** 対象は、このランタイムが完了まで観測できた試行です。ソース更新とローカルコミットの間でプロセスが死ねば、編集も監査エントリも失われます。この隙間を塞ぐには pending-invocation の永続化された記録が必要ですが、本実装は持っていません。

書き込み経路の内側でのクラッシュは **`EXECUTION_CRASHED`** として監査されます。ストレージ障害、またはモデルコード（可視性述語、事前条件、効果関数）が throw した場合で、エラーはその後呼び出し側へ伝播します。

監査の書き込み自体が、失敗しうる箇所であってはなりません。JSON に直列化して復元すると値が変わる params は、モデルが走る前に **`INVALID_PARAMS`** として拒否されます。それでもログにエンコードできない値が残った場合は `$unserializable` プレースホルダとして記録します。欠損した監査エントリより、劣化した監査エントリの方がましだからです。

## トランザクションの所有権

強制されるルールは 1 つだけです。呼び出し側はランタイムをトランザクションで包めません。`execute()` と `load()` は、呼び出し側が開いたトランザクションの中では実行を拒否します。その中では「コミット済み」が実際には「呼び出し側が savepoint をロールバックするまで」を意味してしまい、適用と監査を終えたアクションが、成功報告の後に取り消せてしまうからです。これは原子性の保証であって、侵入への防御ではありません。

境界の残りは、防御ではなく宣言です。このランタイムはプロセス内ライブラリなので、データベースハンドルを握っているコード（呼び出し側、ルール、write-back アダプタ）は、直接 `UPDATE` を発行すればアクションゲートを迂回できます。プロセス内の検査でこれを防ぐことはできません。そこで契約として、ルールとアダプタはオントロジーストアに触れてはならない、と定めています。アダプタには触れる理由がそもそもありません。編集プランとターゲットオブジェクトの自分専用コピーを受け取り、話す相手はシステムオブレコードだけだからです。

以前の版は、この契約違反のうち観測可能な一部分（ストアに開いたまま残されたトランザクション）を検知していましたが、v0.2 で削除しました。侵入の一形態しか捕まえられず、最も単純な形（autocommit での直接 `UPDATE`）を素通しする検査は、防御のように見えて防御ではないからです。強制力のある境界が必要な配備では、ランタイムをプロセス境界の向こう側に置き、消費者がデータベースへ直接アクセスできない構成にしてください。付属の MCP サーバーは、まさにその構成です。

## 保存できる値の境界

ストアは JSON を保持します。したがって、保存されるすべての値は、JSON への直列化と復元を無変更で通過しなければなりません。復元時に姿を変えるか脱落する値、たとえばクラスインスタンス、`Date`、`NaN`、`Map`、任意の深さの `undefined` を含む行は、アクション経由でも `load()` 経由でも、書き込みのたびに拒否されます。同じ検査を、`owned` プロパティの宣言済みデフォルトは定義時に、アクションの params は入口で受けます（前述の監査の項を参照）。

1 つだけ、検査ではなく宣言に留めている義務があります。プロパティスキーマには、値を変換しない検証専用のスキーマを使ってください。ランタイムは保存済みの値を後の書き込みで同じスキーマに再度通すため、変換を行うスキーマ（`z.coerce.date()` や `.transform(…)`）は、次回の検証で自分自身の出力を拒否するか、気づかれないまま書き換えてしまいます。以前の版はこれを書き込みごとの不動点検査（検証結果をもう一度検証し、一致を要求する）で強制していましたが、v0.2 ではモデル作者が守る契約として明文化する方を選びました。

## 再インデックスと編集

スナップショットのセマンティクスは、ロードされた型ごとに「ベースを置き換え、編集レイヤーを再適用する」です。個々の規則を、結果から先に示します。

- **拒否：ontology-owned な状態を供給するスナップショット行。** ontology-owned なプロパティに値を設定する行、ontology-owned な型の行、ontology-owned なリンクのインスタンスはすべて拒否されます。ソースはそのいずれの所有者でもないからです。
- **保持：ontology-owned なプロパティへの編集。** 編集は (type, pk) をキーとするオーバーレイに置かれ、再インデックス後、現在のパッチが新しいベースの上に再適用されます。宣言済みデフォルトへ戻された編集はオーバーレイから取り除かれます。編集を消すことは、それを保持する義務も消すということです（比較は構造的に行われるため、キーの順序で「デフォルトに戻った」ことを偽装することも隠すこともできません）。
- **そのまま保持：ontology-owned な型とリンク。** `load()` はそれらに触れること自体を拒否するので、オーバーレイは不要です。
- **丸ごと拒否：編集を取り残すロード。** ontology-owned な編集を抱えたベース行が消えるようなロードは全体が拒否され、以前の状態がそのまま残ります。その状態をどうするかは照合上の判断であり、ランタイムは照合上の判断を黙って下しません。編集をクリアするか行を復元してから、再ロードしてください。編集のクリア自体もアクションなので、照合作業も書き込みゲートの内側に留まります。
- **拒否：モデルがもう所有しないオーバーレイキー。** モデルが ontology-owned と宣言しなくなったプロパティをオーバーレイのパッチがまだ持っている場合、ロードは拒否されます。その状態の行く末を決めるのは明示的なスキーマ進化であって、リフレッシュではないからです。
- **拒否：制約を壊す部分スナップショット。** ロード対象外の型に残っている状態がモデルの制約に違反する場合、再インデックス全体が拒否され、ロールバックされます。
